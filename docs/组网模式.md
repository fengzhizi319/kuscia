
#  1 组网模式

Kuscia 提供了灵活的组网方案，可以根据不同的业务场景和组织架构选择合适的部署模式。

| 组网模式 | 控制平面 | 节点类型 | 适用场景 |
|----------|----------|----------|----------|
| **中心化组网** | 多个 Lite 节点共享一个 Master | Lite 节点 | 大型机构内部，统一运维，资源成本低 |
| **P2P 组网** | 每个节点自带独立控制平面 | Autonomy 节点 | 小型机构、安全性要求高 |
| **混合组网** | Master + Autonomy 混合部署 | Lite + Autonomy | 复杂组织架构，部分集中部分对等 |
| **K8s 部署** | 外部 K8s 集群提供控制能力 | K8s Deployment | 已有成熟 K8s 基础设施的机构 |
| **互联互通** | 跨平台协议适配 | InterConn Adapter | 与第三方隐私计算平台协作 |

Kuscia 还支持 **中心化与 P2P 混合组网**，以及 **与第三方厂商节点互联互通**。

#  2 节点角色

| 角色 | 说明 | 运行模式 |
|------|------|----------|
| **Master** | 控制平面，负责任务调度、资源管理、路由控制 | `RunModeMaster` |
| **Lite** | 无控制平面的工作节点，依赖 Master 调度 | `RunModeLite` |
| **Autonomy** | 自带控制平面的 P2P 节点 | `RunModeAutonomy` |

#  3 基础设施部署方式

| 部署方式 | 说明 | 适用场景 |
|----------|------|----------|
| **Docker 模式** | 以 Docker 容器方式部署控制平面和节点 | 物理机/虚拟机 |
| **K8s 模式** | 将控制平面和节点以 K8s 应用方式部署 | 公有 K8s 集群 |
| **K8s 控制器模式** | 将 Kuscia Controllers、Storage、Envoy 部署在 K8s 控制平面 | 专有 K8s 集群 |

#  4 运行模式核心差异

三种运行模式并不是简单地在配置文件中切换一个字段，而是直接决定了 Kuscia 会启动哪些内部模块、是否嵌入 K3s、KubeClient 如何初始化、以及节点在联邦中扮演的角色。下面从模块组合、职责划分和配置文件三个维度进行系统对比。

##  4.1 模块启动矩阵

Kuscia 的模块注册集中在 `cmd/kuscia/start/start.go` 中。`ModuleManager.Regist` 的第三个参数开始是可变长的模式白名单，只有当前运行模式出现在白名单中的模块才会被启动。下表汇总了主要模块与模式的对应关系。

| 模块 | Master | Lite | Autonomy | 关键职责 |
|------|--------|------|----------|----------|
| **k3s** | ✅ | ❌ | ✅ | 嵌入 K3s 作为控制平面，提供 Kubernetes API |
| **agent** | ❌ | ✅ | ✅ | Kubelet 替代，管理本地容器生命周期 |
| **containerd** | ❌ | ✅* | ✅* | 本地容器运行时，仅在 `EnableContainerd=true` 时启动 |
| **controllers** | ✅ | ❌ | ✅ | 运行 Kuscia 自定义控制器（Domain、Task、Job 等 CRD） |
| **coredns** | ✅ | ✅ | ✅ | 为 K3s 集群与 Pod 提供 DNS 服务 |
| **scheduler** | ✅ | ❌ | ✅ | 运行增强版 kube-scheduler，支持 Kuscia 调度策略 |
| **datamesh** | ❌ | ✅ | ✅ | 提供数据访问代理与 Arrow Flight 服务 |
| **transport** | ❌ | ✅ | ✅ | 提供节点间安全通信通道 |
| **domainroute** | ✅ | ✅ | ✅ | 维护跨域路由与证书交换 |
| **interconn** | ✅ | ❌ | ✅ | 互联互通协议适配器 |
| **reporter** | ✅ | ❌ | ✅ | 负责节点任务状态上报/采集 |
| **kusciaapi** | ✅ | ✅ | ✅ | 对外暴露 Kuscia API，非 Master 模式下 Initiator 默认为 DomainID |
| **envoy** | ✅ | ✅ | ✅ | 七层网关，代理跨域与域内流量 |
| **confmanager** | ✅ | ✅ | ✅ | 配置中心，Lite 模式下复用 Master 签发的域证书 |
| **metricexporter/nodeexporter/ssexporter/diagnose** | ✅ | ✅ | ✅ | 监控、节点指标、系统状态、诊断等可观测模块 |

> *注：containerd 是否启动取决于运行时配置；若使用外部容器运行时则不会注册。*

从上表可以清晰看出：

- **Master** 只承担控制平面职责：运行 K3s、controllers、scheduler、reporter 等。
- **Lite** 只承担工作节点职责：运行 agent、datamesh、transport 等，通过 `masterEndpoint` 连接到远程 Master 的 K3s API。
- **Autonomy** 是 Master 与 Lite 的合体：既嵌入 K3s 控制平面，又运行 agent 等数据平面模块，因此具备独立对外提供联邦协作的能力（P2P 节点）。

##  4.2 控制平面与数据平面职责划分

| 维度 | Master | Lite | Autonomy |
|------|--------|------|----------|
| **是否嵌入 K3s** | 是 | 否 | 是 |
| **K8s API 来源** | 本地 K3s | 远程 Master | 本地 K3s |
| **是否运行 Agent** | 否 | 是 | 是 |
| **是否运行 Scheduler** | 是 | 否 | 是 |
| **是否运行 Controllers** | 是 | 否 | 是 |
| **域证书来源** | 自签 CA | 由 Master 注册时签发 | 自签 CA |
| **典型部署位置** | 中心集群 | 参与方边缘节点 | 独立参与方全栈节点 |

##  4.3 配置文件结构差异

Kuscia 为三种模式定义了独立的配置结构体，位于 `cmd/kuscia/confloader/kuscia_config.go`。

**Master 配置（`MasterKusciaConfig`）**

```go
type MasterKusciaConfig struct {
CommonConfig `yaml:",inline"`
DatastoreEndpoint string
ClusterToken      string
}
```

关键字段：`DatastoreEndpoint` 指定 etcd/Datastore 地址；`ClusterToken` 用于 K3s 集群加入认证。

**Lite 配置（`LiteKusciaConfig`）**

```go
type LiteKusciaConfig struct {
CommonConfig `yaml:",inline"`
LiteDeployToken   string
MasterEndpoint    string
Runtime           string
Runk              RunkConfig
Capacity          config.CapacityCfg
ReservedResources config.ReservedResourcesCfg
Image             ImageConfig
}
```

关键字段：`MasterEndpoint` 指向远程 Master 的 API 入口；`Runtime`/`Runk` 指定本地容器运行时；`Capacity` 与 `ReservedResources` 描述节点资源。

**Autonomy 配置（`AutonomyKusciaConfig`）**

```go
type AutonomyKusciaConfig struct {
CommonConfig `yaml:",inline"`
Runtime           string
Runk              RunkConfig
Capacity          config.CapacityCfg
ReservedResources config.ReservedResourcesCfg
Image             ImageConfig
DatastoreEndpoint string
}
```

Autonomy 同时具备控制平面与工作节点属性，因此既有 `DatastoreEndpoint` 这类控制平面字段，也包含 `Runtime`、`Capacity` 等节点字段。

对应的默认初始化逻辑也体现了差异：

- `defaultMaster`：生成 K3s API Server 监听地址、kubeconfig 路径等。
- `defaultLite`：仅初始化 agent 相关配置，不生成 K3s 配置。
- `defaultAutonomy`：合并 `defaultMaster` 与 Lite 的 agent 配置。

#  5 实现代码层面的差异

##  5.1 模块注册差异（`cmd/kuscia/start/start.go`）

代码中通过 `mm.Regist` 显式声明每个模块支持的模式。`cmd/kuscia/start/start.go` 中的实际注册如下（已省略依赖设置与就绪钩子）：

```go
master, lite, autonomy := common.RunModeMaster, common.RunModeLite, common.RunModeAutonomy

mm.Regist("coredns", modules.NewCoreDNS, autonomy, lite, master)
mm.Regist("k3s", modules.NewK3s, autonomy, master)
mm.Regist("agent", modules.NewAgent, autonomy, lite)
mm.Regist("envoy", modules.NewEnvoy, autonomy, lite, master)
if conf.EnableContainerd {
mm.Regist("containerd", modules.NewContainerd, autonomy, lite)
}
mm.Regist("config", modules.NewConfManager, autonomy, lite, master)
mm.Regist("controllers", modules.NewControllersModule, autonomy, master)
mm.Regist("datamesh", modules.NewDataMesh, autonomy, lite)
mm.Regist("domainroute", modules.NewDomainRoute, autonomy, master, lite)
mm.Regist("interconn", modules.NewInterConn, autonomy, master)
mm.Regist("kusciaapi", modules.NewKusciaAPI, autonomy, lite, master)
mm.Regist("metricexporter", modules.NewMetricExporter, autonomy, lite, master)
mm.Regist("nodeexporter", modules.NewNodeExporter, autonomy, lite, master)
mm.Regist("ssexporter", modules.NewSsExporter, autonomy, lite, master)
mm.Regist("scheduler", modules.NewScheduler, autonomy, master)
mm.Regist("transport", modules.NewTransport, autonomy, lite)
mm.Regist("reporter", modules.NewReporter, autonomy, master)
mm.Regist("diagnose", modules.NewDiagnose, autonomy, lite, master)
```

`ModuleManager.Regist` 的第三个参数开始是可变长的模式白名单；如果当前运行模式不在白名单中，该模块不会被实例化。例如 Lite 节点不会启动 `k3s`、`controllers`、`scheduler`、`reporter`、`interconn`，因此进程数更少、资源占用更低。第 4.1 节还会进一步说明模块间的依赖关系（如 `envoy -> k3s`、`agent -> envoy/k3s/kusciaapi` 等）。

##  5.2 运行时配置初始化差异（`cmd/kuscia/modules/runtime.go`）

`BuildModuleRuntimeConfigs` 在读取 `kuscia.yaml` 后，会按模式填充 `ModuleRuntimeConfigs`。关键差异如下。

**Master / Autonomy**：

```go
if dependencies.RunMode == common.RunModeMaster || dependencies.RunMode == common.RunModeAutonomy {
dependencies.ApiserverEndpoint = dependencies.Master.APIServer.Endpoint
dependencies.KubeconfigFile = dependencies.Master.APIServer.KubeConfig
dependencies.KusciaKubeConfig = filepath.Join(dependencies.RootDir, "etc/kuscia.kubeconfig")
dependencies.InterConnSchedulerPort = defaultInterConnSchedulerPort
}
```

K3s 模块随后会使用 `KubeconfigFile` 启动本地 API Server；controllers、scheduler 等模块再通过该 kubeconfig 连接本地 K3s。`KusciaKubeConfig` 用于生成面向 Kuscia 内部组件的 kubeconfig，`InterConnSchedulerPort` 则在 Master/Autonomy 上启用互联互通调度端口。

**Autonomy / Lite（工作节点属性）**：

```go
if dependencies.RunMode == common.RunModeAutonomy || dependencies.RunMode == common.RunModeLite {
dependencies.ContainerdSock = common.ContainerdSocket()
dependencies.TransportConfigFile = filepath.Join(dependencies.RootDir, "etc/conf/transport/transport.yaml")
dependencies.TransportPort, err = GetTransportPort(...)
}
```

这两类模式需要本地容器运行时与跨节点通信通道，因此会初始化 containerd socket 与 transport 配置。

**Lite 独有逻辑**：

```go
if dependencies.RunMode == common.RunModeLite {
dependencies.ApiserverEndpoint = defaultEndpointForLite
clients, err := kubeconfig.CreateClientSetsFromKubeconfig(
dependencies.KubeconfigFile,
dependencies.ApiserverEndpoint,
)
dependencies.Clients = clients
}
```

Lite 模式没有本地 K3s，因此直接使用 Master 提供的 kubeconfig 创建到远程 API Server 的 Kubernetes 客户端，并把 `ApiserverEndpoint` 固定为面向 Lite 的默认值。

##  5.3 K3s 启动差异（`cmd/kuscia/modules/k3s.go`）

`k3s` 模块仅存在于 Master/Autonomy。`Run` 方法会启动精简版 K3s server：

```go
args := []string{
"server",
"-d=" + s.dataDir,
"-o=" + s.kubeconfigFile,
"--disable-agent",
"--disable-scheduler",
"--flannel-backend=none",
"--disable=coredns",
"--disable=traefik",
"--disable=servicelb",
// ...
}
```

注意 Kuscia 关掉了 K3s 自带的 agent、scheduler、flannel、coredns、traefik、servicelb 等组件，只保留 API Server 与数据存储，并复用自己的 scheduler、agent（仅 Autonomy）和网络代理。

##  5.4 各模块内部的模式判断

即使某些模块在三种模式下都会启动，其内部行为也会根据 `RunMode` 做分支。

**KusciaAPI（`cmd/kuscia/modules/kusciaapi.go`）**

```go
if d.RunMode != common.RunModeMaster {
kusciaAPIConfig.Initiator = d.DomainID
}
```

在非 Master 模式下，KusciaAPI 的默认发起方被设置为当前节点 DomainID，因为 Lite/Autonomy 节点通常以自身身份提交任务。

**ConfManager（`cmd/kuscia/modules/confmanager.go`）**

```go
switch d.RunMode {
case common.RunModeLite:
conf.DomainCertValue = &d.DomainCertByMasterValue
case common.RunModeAutonomy:
conf.DomainCertValue = &atomic.Value{}
conf.DomainCertValue.Store(d.DomainCert)
}
```

Lite 节点的域证书由 Master 在注册时签发并保存到 `DomainCertByMasterValue`；Autonomy 则使用本地自签 CA 生成的域证书。

**DomainRoute（`cmd/kuscia/modules/domainroute.go`）**

```go
func getPubkeyForToken(pubkey string, runmode common.RunModeType) (*rsa.PublicKey, error) {
if pubkey == "" && runmode == common.RunModeLite {
err := fmt.Errorf("query peer master pubkey failed, pubkey is empty")
return nil, err
}
if runmode == common.RunModeLite {
masterDer, decodeErr := base64.StdEncoding.DecodeString(pubkey)
// ...
return pubKey, nil
}
return nil, nil
}
```

Lite 节点必须获取对端 Master 的公钥用于 UID-RSA Token 校验；Autonomy/Master 则不需要此流程。

#  6 跨域组合与管理机制

Kuscia 支持多种参与方组合方式，通过声明式 API 和控制器调和机制实现统一的跨域管理。本节详细说明不同组网模式下的节点如何发现、注册、认证和通信。

##  6.1 核心概念

在深入组网细节前，需要理解以下关键概念：

| 概念 | 说明 |
|------|------|
| **Domain（域）** | 每个参与方的唯一标识，如 `alice`、`bob`，对应一个 Kubernetes Namespace |
| **Deploy Token** | 用于节点注册的临时令牌，由 Master/Autonomy 生成，一次性使用，存储在 `Domain.Status.DeployTokenStatuses` |
| **DomainRoute** | 命名空间级别的跨域路由配置，包含证书、Token、心跳等状态 |
| **ClusterDomainRoute** | 集群级别的跨域路由，自动同步到源域和目标域的 Namespace |
| **Handshake（握手）** | 双向认证过程，交换加密 Token 并建立安全通信通道 |
| **Revision Token** | 握手后生成的会话 Token，用于后续所有跨域请求的鉴权 |

##  6.2 P2P 组网（Autonomy 模式）的组合与管理

### 场景描述

多个参与方（如 Alice、Bob、Charlie）各自独立部署完整的 Autonomy 节点，彼此对等，没有中心控制平面。

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Alice Node  │◄────┤   Bob Node   │────►│ Charlie Node │
│  Autonomy    │     │  Autonomy    │     │  Autonomy    │
│  - K3s       │     │  - K3s       │     │  - K3s       │
│  - Agent     │     │  - Agent     │     │  - Agent     │
│  - CA        │     │  - CA        │     │  - CA        │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 初始化阶段

每个参与方独立启动自己的 Autonomy 节点：

```bash
# Alice 节点
./kuscia.sh autonomy --domain-id=alice --runtime=runc

# Bob 节点  
./kuscia.sh autonomy --domain-id=bob --runtime=runc

# Charlie 节点
./kuscia.sh autonomy --domain-id=charlie --runtime=runc
```

**每个节点启动时：**
- ✅ 嵌入 K3s（本地 API Server + etcd）
- ✅ 运行 Controllers（Domain、Task、Job 控制器）
- ✅ 运行 Agent（管理本地容器）
- ✅ 运行 Scheduler（调度任务）
- ✅ 自签 CA 证书，生成本地域证书

此时，**各节点完全独立，互不相识**。

### 跨域注册流程（以 Alice 连接 Bob 为例）

#### 步骤 1：创建 Domain CRD

在 **Alice 的 K3s** 中手动创建 Bob 的 Domain 对象：

```yaml
# alice-node: kubectl apply -f bob-domain.yaml
apiVersion: kuscia.secretflow/v1alpha1
kind: Domain
metadata:
  name: bob
spec:
  cert: ""  # 初始为空，等待注册后填充
  authCenter:
    tokenGenMethod: UID-RSA  # 认证方式
```

同样，**Bob 也需要创建 Alice 的 Domain 对象**。

#### 步骤 2：生成 Deploy Token

在 **Alice 节点**，Domain Controller 会自动为 `bob` Domain 生成 Deploy Token：

```go
// pkg/controllers/domain/deploy_token.go
func (c *Controller) generateTokenStatus(size int) []kusciaapisv1alpha1.DeployTokenStatus {
    tokens := make([]kusciaapisv1alpha1.DeployTokenStatus, 0)
    for i := 0; i < size; i++ {
        tokens = append(tokens, kusciaapisv1alpha1.DeployTokenStatus{
            Token: generateToken(),  // 随机32字节
            State: pkgcommon.DeployTokenUnusedState,
        })
    }
    return tokens
}
```

生成的 Token 存储在 `Domain.Status.DeployTokenStatuses` 中，状态为 `unused`。

#### 步骤 3：配置 ClusterDomainRoute

在 **Alice 节点**创建 ClusterDomainRoute：

```yaml
# alice-node: kubectl apply -f alice-to-bob-cdr.yaml
apiVersion: kuscia.secretflow/v1alpha1
kind: ClusterDomainRoute
metadata:
  name: alice-to-bob
spec:
  source: alice
  destination: bob
  endpoints:
    - host: bob-gateway.example.com
      port: 8082
  tokenConfig:
    genMethod: UID-RSA
```

**ClusterDomainRoute Controller** 会自动：
1. 在 `alice` namespace 下创建对应的 `DomainRoute` 对象
2. DomainRoute Module 开始尝试与 Bob 建立连接

#### 步骤 4：发起注册请求（Register）

Alice 的 **DomainRoute Module** 执行 [`RegisterDomain`](file://\\wsl.localhost\Ubuntu\home\charles\code\sfwork\kuscia\pkg\gateway\controller\register_node.go#L57-L81)：

```go
func RegisterDomain(namespace, path, csrData string, priKey *rsa.PrivateKey, afterRegisterHook AfterRegisterDomainHook) error {
    // 1. 生成 JWT Token（用 Alice 私钥签名）
    req, token, err := generateJwtToken(namespace, csrData, priKey)
    
    // 2. 向 Bob 的 Handshake 服务发送注册请求
    err = doHTTPWithDefaultRetry(req, regResp, &utils.HTTPParam{
        Method:       http.MethodPost,
        Path:         fmt.Sprintf("%s/register", path),
        KusciaSource: namespace,  // alice
        KusciaHost:   "bob.master.svc",  // Bob 的 Handshake 端点
    })
    
    // 3. 收到响应后，保存 Bob 颁发的证书和公钥
    afterRegisterHook(regResp)
    return nil
}
```

**CSR 数据结构：**
```go
req = &handshake.RegisterRequest{
    DomainId:    "alice",
    Csr:         base64.StdEncoding.EncodeToString(csrData),  // PEM 格式 CSR
    RequestTime: time.Now().Nanosecond(),
}
```

CSR 中包含：
- CN = `alice`（DomainID）
- Deploy Token（放在扩展字段 `1.2.3.4`）
- Alice 的公钥

#### 步骤 5：Bob 验证并颁发证书

Bob 节点的 [`registerHandle`](file://\\wsl.localhost\Ubuntu\home\charles\code\sfwork\kuscia\pkg\gateway\controller\register_node.go#L107-L218) 处理注册：

```go
func (c *DomainRouteController) registerHandle(w http.ResponseWriter, r *http.Request) {
    // 1. 解析 CSR
    certRequest, err := parseCertRequest(req.Csr)
    
    // 2. 验证 JWT Token（用 Alice 公钥验签）
    err = verifyRegisterRequest(&req, r.Header.Get("jwt-token"))
    
    // 3. 从 CSR 中提取 Deploy Token 并验证
    index, matchErr := deployTokenMatched(certRequest, domain.Status.DeployTokenStatuses)
    // - 检查 Token 是否存在
    // - 检查 Token 状态是否为 unused
    
    // 4. 用 Bob 的 CA 为 Alice 签发证书
    domainCrtRaw, err := x509.CreateCertificate(rand.Reader, domainCrt, c.CaCert, 
                                                certRequest.PublicKey, c.CaKey)
    
    // 5. 更新 Domain 状态
    domainDeepCopy.Spec.Cert = domainCrtStr  // 存储 Alice 的证书
    domainDeepCopy.Status.DeployTokenStatuses[index].State = common.DeployTokenUsedState
    
    // 6. 返回证书和 Bob 的公钥
    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(&handshake.RegisterResponse{
        Cert:         domainCrtStr,
        MasterPubkey: base64.StdEncoding.EncodeToString(tls.EncodePKCS1PublicKey(c.prikey)),
    })
}
```

#### 步骤 6：握手（Handshake）

注册成功后，Alice 执行 [`HandshakeToMaster`](file://\\wsl.localhost\Ubuntu\home\charles\code\sfwork\kuscia\pkg\gateway\controller\handshake.go#L742-L799)：

```go
func HandshakeToMaster(domainID string, pathPrefix string, prikey *rsa.PrivateKey, masterPubKey *rsa.PublicKey) (*RevisionToken, error) {
    // 1. 准备握手请求
    handshakeReq, sourceToken, err := prepareHandshakeReq(domainID, pathPrefix, prikey, masterPubKey)
    // - 生成随机 Token（源 Token）
    // - 用 Bob 公钥加密源 Token
    // - 计算 Alice 公钥哈希
    
    // 2. 发送握手请求到 Bob
    resp := &handshake.HandShakeResponse{}
    utils.DoHTTP(handshakeReq, resp, &utils.HTTPParam{
        Path: "/handshake",
        KusciaHost: "bob.master.svc",
    })
    
    // 3. 解密 Bob 返回的 Token（目标 Token）
    token, decryptErr := decryptToken(prikey, resp.Token.Token, tokenByteSize/2)
    
    // 4. 拼接完整 Token：sourceToken + targetToken
    token = append(sourceToken, token...)
    
    // 5. 将 Token 添加到 Envoy 集群配置
    addMasterClusters(domainID, pathPrefix, token)
    
    return &RevisionToken{
        RawToken: token,
        Revision: resp.Token.Revision,
    }
}
```

**UID-RSA Token 生成机制：**

```
Alice                          Bob
─────                          ───
1. 生成 random_A（16字节）
2. 用 Bob 公钥加密 random_A
3. 发送加密的 random_A  ──→  4. 用私钥解密得到 random_A
                               5. 生成 random_B（16字节）
                               6. 用 Alice 公钥加密 random_B
7. 接收加密的 random_B  ←──  8. 发送加密的 random_B
9. 用私钥解密得到 random_B
10. 最终 Token = random_A + random_B（32字节）
```

**握手结果：**
- Alice 获得双向认证的 Token
- Token 被注入到 Envoy 的 HTTP Header（`jwt-token`）
- 后续所有跨域请求都携带此 Token

#### 步骤 7：双向同步

上述流程是 **单向的**（Alice → Bob）。要实现双向通信，Bob 也需要执行相同流程连接 Alice：

```
Alice Node                          Bob Node
─────────                           ────────
1. 创建 Bob Domain              ←→  1. 创建 Alice Domain
2. 生成 Deploy Token for Bob    ←→  2. 生成 Deploy Token for Alice
3. 创建 ClusterDomainRoute      ←→  3. 创建 ClusterDomainRoute
4. Register to Bob              ←→  4. Register to Alice
5. Handshake with Bob           ←→  5. Handshake with Alice
```

##  6.4 K8s 部署模式的组合与管理

### 场景描述

K8s 部署模式是指将 Kuscia 的控制平面和节点以 Kubernetes 应用（Deployment/Service）的方式部署到**外部已有的 K8s 集群**中，而不是使用嵌入式的 K3s。这种模式适合已经有成熟 K8s 运维体系的机构。

**两种子模式：**

1. **K8s 标准模式**：Kuscia Controllers、Storage、Envoy 作为独立 Pod 运行在 K8s 控制平面
2. **K8s 控制器模式**：仅部署 Kuscia CRD 控制器，复用现有 K8s 控制平面能力

```
┌──────────────────────────────────────────────┐
│         外部 K8s 集群（已有）                  │
│                                              │
│  ┌─────────────┐    ┌─────────────┐          │
│  │ Master Pod  │    │  Lite Pod   │          │
│  │ Deployment  │    │ Deployment  │          │
│  │ - kuscia    │    │ - kuscia    │          │
│  │   kernel    │    │   kernel    │          │
│  │ - envoy     │    │ - envoy     │          │
│  └──────┬──────┘    └──────┬──────┘          │
│         │                 │                  │
│         ▼                 ▼                  │
│  ┌─────────────────────────────┐             │
│  │  K8s API Server (外部)      │             │
│  │  - 提供调度能力              │             │
│  │  - 提供存储能力              │             │
│  └─────────────────────────────┘             │
└──────────────────────────────────────────────┘
```

### 与嵌入式 K3s 模式的核心差异

| 维度 | 嵌入式 K3s（Master/Autonomy/Lite） | K8s 部署模式 |
|------|-----------------------------------|-------------|
| **API Server 来源** | 内置 K3s API Server | 外部 K8s API Server |
| **数据存储** | K3s 内置 etcd/Datastore | MySQL/PostgreSQL/外部 etcd |
| **调度器** | 自研 Scheduler 模块 | 复用 K8s 原生调度器 |
| **网络插件** | 无（禁用 Flannel） | 复用 K8s CNI（Calico/Flannel等） |
| **证书管理** | 自签 CA | 可集成 K8s cert-manager |
| **资源隔离** | Namespace 级别 | Namespace + ResourceQuota |
| **运维复杂度** | 低（一键启动） | 高（需维护 K8s 集群） |

✅ **资源调度更灵活**：利用 K8s 原生调度器，支持复杂调度策略

##  6.6 互联互通模式（Interop）

### 场景描述

互联互通（InterConn）是指 Kuscia 与**第三方隐私计算平台**（如 FATE、OpenMMLab 等）进行跨域协作的能力。通过标准化的协议适配层，实现不同厂商系统之间的任务协同。

```
┌──────────────────┐              ┌──────────────────┐
│   Kuscia Node    │              │  Third-Party     │
│   (Alice)        │◄════════════►│  Platform        │
│                  │  InterConn   │  (Bob - FATE)    │
│  - Controllers   │   Protocol   │                  │
│  - InterConn     │              │  - Adapter       │
│    Adapter       │              │  - Scheduler     │
│  - Envoy         │              │  - Gateway       │
└──────────────────┘              └──────────────────┘
```

### 核心架构组件

#### InterConn 模块

[`InterConn`](file://\\wsl.localhost\Ubuntu\home\charles\code\sfwork\kuscia\cmd\kuscia\modules\interconn.go) 是 Kuscia 的互联互通协议适配器，仅在 **Master** 和 **Autonomy** 模式下启动：

```go
// cmd/kuscia/start/start.go
mm.Regist("interconn", modules.NewInterConn, autonomy, master)
```

**关键职责：**
1. **协议转换**：将 Kuscia 内部的任务描述转换为第三方平台可理解的格式
2. **任务映射**：维护 Kuscia Task 与第三方 Job 的对应关系
3. **状态同步**：监听第三方平台任务状态并回报到 Kuscia CRD
4. **数据桥接**：协调跨平台的数据访问权限和传输

#### InterConnScheduler

独立的调度服务，提供 HTTP/gRPC API 供第三方平台调用：

```go
// cmd/kuscia/modules/runtime.go
dependencies.InterConnSchedulerPort = defaultInterConnSchedulerPort  // 默认 9090
```

**API 端点：**
- `POST /api/v1/interconn/schedule`: 提交跨域任务
- `GET /api/v1/interconn/status/{job_id}`: 查询任务状态
- `POST /api/v1/interconn/cancel/{job_id}`: 取消任务

### 互联互通注册流程

#### 步骤 1：在第三方平台注册 Kuscia 节点

以 FATE 为例，需要在 FATE Flow 中配置 Kuscia 作为远程计算引擎：

```json
// fate_flow config
{
  "computing_engine": "KUSCIA",
  "kuscia_config": {
    "endpoint": "http://alice-gateway.example.com:1080",
    "domain_id": "alice",
    "auth_token": "shared_secret_token"
  }
}
```

#### 步骤 2：在 Kuscia 中配置第三方 Domain

在 **Kuscia Master/Autonomy** 中创建第三方平台的 Domain：

```yaml
apiVersion: kuscia.secretflow/v1alpha1
kind: Domain
metadata:
  name: bob-fate  # 第三方平台的标识
spec:
  cert: ""  # 初始为空
  authCenter:
    tokenGenMethod: UID-RSA
  interopConfig:
    protocol: FATE  # 或 OPENMMLAB、PRIVACYWARE 等
    adapterEndpoint: http://bob-fate-gateway.example.com:9380
```

**关键字段：**
- `interopConfig.protocol`: 第三方平台类型
- `interopConfig.adapterEndpoint`: 第三方平台的适配器地址

#### 步骤 3：配置 ClusterDomainRoute

创建从 Kuscia 到第三方平台的跨域路由：

```yaml
apiVersion: kuscia.secretflow/v1alpha1
kind: ClusterDomainRoute
metadata:
  name: alice-to-bob-fate
spec:
  source: alice
  destination: bob-fate
  endpoints:
    - host: bob-fate-gateway.example.com
      port: 9380
  tokenConfig:
    genMethod: UID-RSA
  interopConfig:
    enabled: true
    protocol: FATE
```

#### 步骤 4：执行 Register 和 Handshake

与标准 P2P 流程相同，但目标端点是第三方平台的适配器：

```bash
# Kuscia 自动执行
kubectl get clusterdomainroute alice-to-bob-fate -o yaml
```

**期望状态：**

```yaml
status:
  conditions:
    - type: Ready
      status: "True"
      reason: Success
  tokenStatus:
    sourceTokens:
      - token: "..."
        isReady: true
    destinationTokens:
      - token: "..."
        isReady: true
```

### 跨平台任务执行流程

#### 示例：Kuscia 发起 PSI 任务到 FATE

##### 1. 用户提交 KusciaJob

```yaml
apiVersion: kuscia.secretflow/v1alpha1
kind: KusciaJob
metadata:
  name: cross-platform-psi
  namespace: alice
spec:
  tasks:
    - name: psi-task
      appImage: secretflow/psi:latest
      parties:
        - domainID: alice
          role: leader
        - domainID: bob-fate  # 第三方平台
          role: follower
```

##### 2. Kuscia Scheduler 调度任务

- 识别 `bob-fate` 为第三方 Domain
- 调用 InterConn Adapter 转换任务描述

##### 3. InterConn 转换任务

```go
// pkg/interconn/adapter/fate_adapter.go
func (a *FateAdapter) ConvertTask(kusciaTask *KusciaTask) (*FateJob, error) {
    return &FateJob{
        JobID: generateFateJobID(kusciaTask.ID),
        DSL:   convertDSL(kusciaTask.Spec.AppImage),
        RuntimeConf: &FateRuntimeConf{
            ComputingEngine: "FATE",
            FederatedMode:   "MULTIPARTY",
        },
    }, nil
}
```

##### 4. 提交到 FATE Flow

```bash
curl -X POST http://bob-fate-gateway.example.com:9380/v1/job/submit \
  -H "Content-Type: application/json" \
  -d '{
    "job_id": "psi-job-xyz",
    "dsl": {...},
    "runtime_conf": {...}
  }'
```

##### 5. 状态同步

InterConn Poller 定期查询 FATE 任务状态并更新 Kuscia CRD：

```go
// pkg/interconn/poller/fate_poller.go
func (p *FatePoller) SyncJobStatus(kusciaTaskID string, fateJobID string) {
    fateStatus := p.queryFateJob(fateJobID)
    kusciaStatus := mapFateStatusToFateStatus(fateStatus)
    
    // 更新 KusciaTask Status
    p.kusciaClient.KusciaV1alpha1().KusciaTasks(namespace).
        UpdateStatus(ctx, &KusciaTask{
            Status: KusciaTaskStatus{
                Phase: kusciaStatus,
            },
        })
}
```

**状态映射表：**

| FATE 状态 | Kuscia 状态 |
|----------|------------|
| running | Running |
| success | Succeeded |
| failed | Failed |
| canceled | Cancelled |

FATE 完成任务后，通过 InterConn 回调接口返回结果：

### 互联互通的安全考虑

#### 1. 双向认证

与标准 P2P 相同，使用 UID-RSA Token 机制确保双方身份可信。

#### 2. 数据隔离

第三方平台只能访问明确授权的 DomainData：

```yaml
apiVersion: kuscia.secretflow/v1alpha1
kind: DomainDataGrant
metadata:
  name: alice-grant-to-bob-fate
  namespace: alice
spec:
  dataID: dataset-001
  granteeDomain: bob-fate
  permissions:
    - READ  # 仅读，不可写
  expireTime: "2024-12-31T23:59:59Z"
```

#### 3. 审计日志

所有跨平台操作都记录审计日志：

```bash
kubectl logs deployment/kuscia-master -n kuscia-master | grep "interop"
```

**日志示例：**

```

[INTEROP] Sync status from FATE: job_id=psi-xyz, status=running
```

### 互联互通的限制与挑战

#### 当前限制

⚠️ **协议兼容性**：不同厂商的协议实现可能存在差异  
⚠️ **功能对等性**：并非所有 Kuscia 功能都能在第三方平台找到对应实现  
⚠️ **性能开销**：协议转换和轮询增加延迟（通常 1-5 秒）  
⚠️ **调试复杂度**：跨系统问题排查需要双方团队协作  

#### 最佳实践

✅ **标准化优先**：优先使用行业标准协议（如 OpenFL）  
✅ **异步解耦**：采用事件驱动架构，避免强依赖第三方平台可用性  
✅ **超时重试**：配置合理的超时和重试策略  
✅ **监控告警**：对 InterConn 健康状态进行监控  

##  6.7 统一管理的关键机制

### 声明式 API

所有配置都通过 Kubernetes CRD 管理：
- `Domain`: 定义参与方
- `DomainRoute`: 定义跨域路由
- `ClusterDomainRoute`: 集群级路由（自动同步）
- `KusciaJob`: 定义联邦任务

### 控制器调和（Reconciliation）

每个模块都有 Controller 持续调和对期望状态：

```go
// ClusterDomainRoute Controller 伪代码
func (c *controller) Reconcile(cdr *ClusterDomainRoute) {
    // 1. 检查源域和目标域的 DomainRoute 是否存在
    srcDR := getDomainRoute(cdr.Spec.Source, cdr.Name)
    dstDR := getDomainRoute(cdr.Spec.Destination, cdr.Name)
    
    // 2. 如果不存在，创建
    if srcDR == nil {
        createDomainRoute(cdr.Spec.Source, cdr)
    }
    
    // 3. 同步状态
    syncStatusFromDomainroute(cdr, srcDR, dstDR)
    
    // 4. 更新条件
    if isReady(srcDR) && isReady(dstDR) {
        setCondition(cdr, Ready, True)
    } else {
        setCondition(cdr, Ready, False, "WaitingForPeer")
    }
}
```

### 证书生命周期管理

```
生成 CSR → 注册（Register） → 颁发证书 → 握手（Handshake） → 定期刷新
```

- **CSR 生成**: 包含 DomainID、公钥、Deploy Token
- **证书颁发**: Master/Autonomy 用自己的 CA 签名
- **握手**: 交换双向认证 Token
- **刷新**: 通过 HeartBeat 维持连接

### Token 认证机制

支持两种模式：

| 模式 | 说明 | 安全性 |
|------|------|--------|
| **UID** | 直接用对方 DomainID 作为 Token | 低（已废弃） |
| **UID-RSA** | 双方各生成随机 Token，拼接后用 RSA 加密 | 高 |

#  7 模式选择建议

| 场景 | 推荐模式 | 理由 |
|------|----------|------|
| 多方协作，由一方统一托管控制平面 | **Master + Lite** | Master 运行中心控制平面，各参与方部署 Lite 节点接入，运维边界清晰 |
| 单个机构独立部署，既要管理又要跑任务 | **Autonomy** | 一台节点即可完成控制与计算，减少部署组件数 |
| 已有外部 Kubernetes 集群，只想添加 Kuscia 能力 | **K8s 模式** | 不依赖 K3s，复用现有 K8s 控制平面 |
| 对资源极度敏感的边缘节点 | **Lite** | 不跑 K3s、scheduler、controllers，内存与 CPU 占用最低 |
| 多机构对等协作，无中心信任方 | **P2P（Autonomy）** | 各节点地位平等，双向认证，去中心化 |
| 复杂组织架构，部分集中部分对等 | **混合模式** | 灵活组合，兼顾集中管理与对等协作 |
| 对接第三方隐私计算平台 | **互联互通（InterConn）** | 标准化协议适配，跨厂商系统协同 |
